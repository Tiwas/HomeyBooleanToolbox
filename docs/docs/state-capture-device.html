State Capture Device

Dynamically capture and restore device states using templates, named slots, and stack operations.

What is a State Capture Device?

While the regular State Device stores fixed values defined during setup, the State Capture Device lets you dynamically capture the current values of devices at runtime.

Think of it as a "template" that defines which devices and capabilities to capture, without storing the actual values until you explicitly capture them.

Key Differences from State Device

Feature State Device State Capture Device
Values Fixed at setup Captured at runtime
Named States One per device Multiple named slots
Stack Operations No Yes (push/pop/peek)
Use Case Predefined scenes Dynamic state management

‚ö† Important: Two Separate Systems

The State Capture Device has two completely separate storage systems that do NOT interact with each other:

üìÅ Named States (Persistent)

Permanent storage with explicit names.

‚Ä¢ Capture state as "name"
‚Ä¢ Apply captured state "name"
‚Ä¢ Delete captured state "name"

States persist until explicitly deleted. Survives restarts.

üìö Stack (Transient)

Temporary LIFO storage for quick save/restore.

‚Ä¢ Push current state to stack
‚Ä¢ Pop and apply from stack
‚Ä¢ Peek (apply without removing)

Designed for temporary operations. No names involved.

These systems are NOT interchangeable!

  • ‚ùå You cannot "pop" a named state
  • ‚ùå You cannot "apply" something from the stack by name
  • ‚ùå You cannot "push" to a named slot

‚ö† Critical: Save Before Use

A state must be saved in a separate flow before you can use it. You cannot save a state and then apply it in the same flow.

‚ùå Will NOT Work

WHEN: Button pressed
THEN: Capture state as "my_scene"
THEN: Apply captured state "my_scene"
       ‚Üë FAILS - state not yet available!

‚úì Correct Approach

FLOW 1 - Save:
WHEN: "Save" button pressed
THEN: Capture state as "my_scene"

FLOW 2 - Apply (later):
WHEN: "Apply" button pressed
THEN: Apply captured state "my_scene"

Why?

The state is saved asynchronously. The "Apply" action in the same flow runs before the save completes, so the state doesn't exist yet.

Use Cases

Temporary Interruptions

Save current lighting state before a doorbell ring, turn on all lights, then restore the previous state after.

WHEN: Doorbell rings
THEN: Push current state
THEN: Set all lights to 100%
THEN: Wait 5 minutes
THEN: Pop state (restore previous)

Named Scene Snapshots

Capture and name the current state when users are happy with their lighting setup.

WHEN: "Save Scene" button pressed
THEN: Capture state as "[[scene_name]]"

WHEN: "Apply Scene" triggered
THEN: Apply captured state "[[scene_name]]"

Undo Functionality

Push state before any major change. Users can "undo" by popping the stack.

Nested State Changes

Stack supports multiple levels - push before each change, pop in reverse order to unwind.

Setting Up a State Capture Device

1

Select Scope

Choose which zones (rooms) contain devices you want to capture. This defines the pool of available devices.

2

Select Devices

Choose which devices from the selected zones should be included in your capture template.

3

Select Capabilities

For each device, select which capabilities to capture (e.g., onoff, dim, light_hue).

Note

Unlike the regular State Device, you don't set values here - only the template of what to capture.

4

Review & Create

Review the generated template configuration and create your device. The template defines what gets captured, not the values.

Flow Cards: Named States

Named states are key-value storage for captured device states. Use dynamic names with Homey tokens.

Action

Action

Retrieves a named state and applies all stored values to the devices.

Uses autocomplete to show existing state names.

State exists

Flow Cards: Stack Operations

The stack provides LIFO (Last-In-First-Out) storage for temporary state management. Perfect for "save and restore" patterns.

Push state to stack

Action

Captures current values and pushes them onto the stack.

Push current state to stack on [State Capture Device]

Pop state from stack

Action

Removes the most recent state from the stack and applies it to devices.

Pop and apply state from stack on [State Capture Device]

Peek and apply state

Action

Applies the most recent state from the stack without removing it.

Clear stack

Action

Removes all states from the stack.

Stack is empty

Condition

Returns true if the stack has no saved states.

Stack depth comparison

Condition

Compare stack depth using operators: equals, greater than, less than.

Storage

States are stored persistently using Homey's app settings. They survive app restarts but are removed when the device is deleted.

Example: Doorbell Interruption

Flow 1: Doorbell Rings

WHEN: Doorbell button pressed
THEN: Push current state to stack [Living Room Lights Capture]
THEN: Set Living Room Lights to 100%
THEN: Wait 5 minutes
THEN: IF stack is NOT empty
THEN:   Pop and apply state from stack [Living Room Lights Capture]

Why use the stack?

If the doorbell rings multiple times, each push adds to the stack. When you pop, you get back to the state just before that specific ring. This handles nested interruptions gracefully.

Error Handling

Device not found

If a device in the template has been removed, it's skipped during capture/apply. Other devices continue normally.

Capability not found

If a capability no longer exists on a device, it's skipped. The error trigger fires with details.

State not found

Applying a non-existent named state or popping an empty stack will fail and trigger the error event.
